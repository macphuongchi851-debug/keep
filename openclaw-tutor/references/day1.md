## 先讲人话：OpenClaw 到底是什么

你可以先把 OpenClaw 理解成：

> 一个把 **模型、工具、文件、聊天入口、自动化** 串起来的个人助手系统。

它不是单纯聊天框。
它更像一个“有脑子、有手、有入口、还能自己定时干活”的助手框架。

一个普通聊天网页，通常只会：
- 你问一句
- 它答一句
- 聊天结束就差不多散了

而 OpenClaw 还能：
- 读写工作区文件
- 跑脚本
- 通过 Telegram/其他渠道接消息
- 记住一部分信息
- 设置定时提醒
- 调浏览器、节点、消息插件等能力

这就是它跟“纯聊天 AI”最大的差别。

---

## 你先记住这 6 个核心词

## 1）Gateway

这是总控中枢。
很多能力都要经过它：
- 消息收发
- 会话管理
- 工具调用
- 状态汇总
- 插件挂载

你可以把它理解成：
**OpenClaw 的心脏 + 交通枢纽。**

Gateway 没起来，很多能力就会一起趴窝。

---

## 2）Session

Session 不是“单条消息”，而是**一段持续对话上下文**。

比如：
- 你在 Telegram 私聊它，是一个 session
- 你在网页里连续聊，也会形成一个 session
- 某个子代理跑任务，也可能有自己的 session

为什么这词重要？
因为很多“它怎么还记得刚才的话”“为什么上下文串了”，本质都跟 session 有关。

---

## 3）Workspace

Workspace 就是默认工作区。

很多文件读写、脚本执行、生成产物，都会围着这里转。
在你这台机器里，当前工作区是：

```text
/home/keep/.openclaw/workspace
```

你可以把它理解成：
**OpenClaw 的办公桌。**

桌上有什么，它就更容易用什么；
桌上文件怎么组织，也会直接影响它干活效率。

---

## 4）Tool

Tool 就是工具能力。

模型本身只会“想”和“说”，
工具让它开始“做事”。

比如：
- 读文件
- 写文件
- 跑命令
- 搜网页
- 控浏览器
- 发消息
- 建定时任务

没有工具，它只是嘴强王者；
有工具，它才像个真助手。

---

## 5）Channel

Channel 就是消息入口。

比如：
- Telegram
- Discord
- Signal
- WhatsApp
- 网页入口

你可以把它理解成：
**别人怎么找到这个助手。**

Channel 解决的是“从哪儿进来”，
Session 解决的是“进来后这段对话怎么算一条线”。

---

## 6）Control UI

Control UI 可以理解成控制面板。

通常用来：
- 看状态
- 看会话
- 做管理动作
- 辅助排障

它不是 OpenClaw 的全部，
但对你理解系统运行状况很有帮助。

---

## 这些东西之间是什么关系

你可以先用这张脑内图：

```text
你 / Telegram / Web
        ↓
     Channel
        ↓
     Gateway
        ↓
 Session + Tools + Workspace
        ↓
   结果返回给你
```

更粗暴一点：

- **Channel**：门
- **Gateway**：中枢
- **Session**：对话线
- **Tool**：手脚
- **Workspace**：办公桌
- **Control UI**：监控屏

这张图装进去，后面很多课你就不容易迷路。

---

## 今天为什么必须先学这个

因为后面你会遇到这些问题：

- 为什么状态异常先看 Gateway？
- 为什么有的聊天不会共享上下文？
- 为什么让它改文件时要先看 workspace？
- 为什么 Telegram 接进来以后还会被策略拦？
- 为什么它“会说不会做”通常是工具没用上？

这些问题如果没地图，只会全糊在一起。

## 今天怎么学：动手但不折腾

今天只做轻量动作，别乱改配置。

### 第一步：看一眼系统状态

执行：

```bash
openclaw status
```

你今天不需要把每个字段都看懂。
你只需要建立一个感觉：

- 这个系统不是“抽象 AI”
- 它是一个**有状态、有组件**的东西

看到状态页，就算今天第一步成功了。

---

### 第二步：确认 workspace 路径

你至少要知道默认工作区在哪。
当前这套环境是：

```text
/home/keep/.openclaw/workspace
```

以后你让它：
- 读文件
- 写文件
- 跑脚本

大多都绕着这里打转。

---

### 第三步：说出这几个概念各是什么

别嫌土，这一步很关键。

你试着用自己的话复述：

- Gateway 是什么
- Session 是什么
- Workspace 是什么
- Tool 是什么
- Channel 是什么

你不用背官方定义。
只要能说清“它大概负责哪摊事”，就够了。

---

## 一个最小例子，帮你把概念串起来

假设你在 Telegram 给 OpenClaw 发一句：

> 帮我读取 workspace 里的某个文件

背后发生的事情大概是：

1. 你从 **Telegram 这个 channel** 发消息
2. 消息被 **Gateway** 接住
3. 进入某个 **session**
4. 它决定调用 **read 工具**
5. 去 **workspace** 里读文件
6. 再把结果通过 channel 回给你

看到没？
这就不是“AI 会聊天”，
这是**系统在处理任务**。

## 今天最容易踩的坑

### 坑 1：把 OpenClaw 当成单纯模型

错。
它是“模型 + 工具 + 状态 +入口 + 文件系统”的组合。

### 坑 2：没搞清层次就开始乱动

这样后面只会：
- 改了配置不知道影响哪
- 报错了不知道从哪查
- 会话串了也不知道为什么

### 坑 3：觉得第一天太虚

别急。
第一天装地图，后面几天你会省很多无意义撞墙。
