# Day 7：Telegram 接入（上）—— 先搞明白“门、钥匙、门禁规则”

## 今天学什么

今天不是立刻把 Telegram 配进去，
而是先把最核心的 3 个东西搞明白：

1. **Bot 是什么**
2. **Token 是什么**
3. **谁可以来找这个 Bot 说话**

如果这 3 个东西没弄清楚，
你后面就很容易把 Telegram 接成一个“能回消息的坑”。

---

## 先讲人话：Telegram 接入到底是在干嘛

你可以把 OpenClaw 想成住在你机器里的助手。
平时你在本地面板里找它，当然可以。

但如果你想：
- 出门也能找它
- 在手机上直接聊它
- 像发消息一样使唤它

那你就得给它开一个“聊天入口”。

**Telegram Bot**，就是这个入口。

---

## 这 3 个东西分别是什么意思

## 1）Bot 是什么

Bot 就是 OpenClaw 在 Telegram 里的“分身账号”。
别人不是直接加 OpenClaw，
而是先加这个 Bot。

你可以理解成：
- OpenClaw = 幕后大脑
- Bot = 对外门脸

---

## 2）Token 是什么

Token 是这个 Bot 的凭证。

说白了就是：
**证明“你真的是这个 Bot 的主人”那把钥匙。**

有了 token，系统才有资格代表这个 Bot 收发消息。

所以 token 很敏感。
别到处贴，别乱发，别截图外传。

---

## 3）访问策略是什么

这决定：
- 谁能给这个 Bot 发消息
- 发了以后系统认不认
- 是先批准、还是直接允许、还是只允许白名单

这一层如果你不设，
Bot 就算接上了，也可能是个安全坑。

---

## 为什么今天不急着上配置

因为很多人一上来就复制一坨配置：
- token 填了
- bot 开了
- restart 了
- 然后发现一堆奇怪问题

本质原因是：
他根本没搞懂“钥匙”和“门禁规则”。

所以今天先把心智模型装好，
明天再进实操。

---

## 你今天要学会什么

1. Telegram Bot 在整个系统里扮演什么角色
2. Token 为什么敏感，为什么不能乱泄露
3. 访问策略为什么跟“能不能聊天”同样重要
4. pairing / allowlist 这类思路分别在解决什么问题

---

## 先讲两种常见访问思路

## 1）Pairing / 审批式

意思是：
有人来找这个 Bot，
但不是自动放行，
而是要先经过批准或配对流程。

适合：
- 更保守
- 不想让陌生人直接接进来
- 想先确认是谁

---

## 2）Allowlist / 白名单式

意思是：
只有明确在允许列表里的用户 ID，才允许接入。

适合：
- 你已经知道只给哪些人用
- 想要直接拦死其他人
- 边界要清楚

这通常比“全开放”靠谱得多。

---

## 今天怎么学

### 第一步：创建或确认 Bot

如果你还没有 Telegram Bot，
通常要先通过 Telegram 官方方式创建一个 Bot，并拿到 token。

这里你今天先记住结果，不需要死背流程：
- 你会得到一个 bot 用户名
- 你会得到一个 token

bot 用户名是门脸，
token 是钥匙。

---

### 第二步：把 token 当密码看待

请直接把 token 当密码。

这意味着：
- 不要发群里
- 不要随手截图
- 不要贴到不可信平台
- 不要让别人轻易拿到

因为一旦 token 泄露，
别人就可能冒用你的 Bot 能力。

---

### 第三步：先决定门禁规则，再谈接入

你至少先问自己：

- 这个 Bot 只给我自己用？
- 还是给少数固定人用？
- 还是暂时调试，后面再收口？

如果只给自己或极少数人用，
**allowlist 往往最直接。**

如果想先人工确认再放行，
**pairing 更稳。**

---

## 一个正确的理解方式

Telegram 接入不是：
“开个 bot，它能回就行。”

而是：
“我开了一个对外入口，我必须知道：
- 谁能进
- 凭什么进
- 进来后怎么管”

这思路一变，后面很多坑就少了。

---

## 今天最容易踩的坑

### 坑 1：把 token 当普通字符串

这本质上是密钥。
别乱搞。

### 坑 2：只在意能不能发消息，不在意谁能接入

这非常危险。

### 坑 3：没区分 bot 用户名和 token

一个是对外可见身份，
一个是控制权凭证。
不是一回事。

### 坑 4：觉得“先全开放，以后再说”

这类“以后再说”通常会变成“以后出事”。

---
