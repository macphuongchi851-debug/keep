# Day 2：启动、状态与基础排障 —— 先确认机器活着，再谈别的

## 今天学什么

今天不追求修大故障，先学一个最重要的习惯：

> **OpenClaw 看起来不对劲时，先看状态，再动手。**

很多人一出问题就去改配置、重启一堆东西、怀疑模型抽风。结果小毛病硬修成火葬场。

今天你学完，至少要做到：
- 会看 `openclaw status`
- 知道出问题时先看哪几行
- 会启动 / 重启 gateway
- 能做最基础的排障

---

## 为什么今天很重要

因为 OpenClaw 不是“永远抽象稳定”的纯网页服务。
它是一个运行中的系统。

系统一旦不对，常见就这几类：
- Gateway 根本没起来
- Gateway 起了，但某个插件/入口异常
- 你以为坏了，其实只是没看状态
- 某个功能挂了，但核心还活着

所以今天的重点不是背命令，
而是建立一个排障顺序。

---

## 先记住这条排障顺序

当你感觉“不对劲”时，默认顺序是：

1. **先看状态**
2. **再判断是哪一层有问题**
3. **必要时重启 Gateway**
4. **重看状态确认有没有恢复**

别反过来。
别一上来瞎改配置。

---

## 今天要学会的 4 件事

1. 跑 `openclaw status`
2. 识别“系统活着没”
3. 学会基础控制命令
4. 出现异常时知道先怎么查

---

## 今天怎么学

### 第一步：看系统状态

执行：

```bash
openclaw status
```

你看到状态后，今天先重点看这些：

- Gateway 是否 running / healthy
- 有没有明显 error / failed / disconnected
- 你关心的通道（比如 Telegram）有没有异常
- 模型/入口/插件有没有直接报红

你今天不用把每个细节都吃透。
重点是学会：

> 出问题时，**状态页是第一现场**。

---

### 第二步：知道几个最常用控制命令

```bash
openclaw gateway status
openclaw gateway start
openclaw gateway stop
openclaw gateway restart
```

这 4 个够你应付很多基础情况了。

先讲人话：
- `status`：看看还活着没
- `start`：没起来就拉起来
- `stop`：先停掉
- `restart`：重新拉一遍，常用于改配置后或状态怪异时

---

### 第三步：学会一个最小排障动作

假设你感觉 OpenClaw 没反应。
那你别慌，按这套来：

#### 步骤 1：先看状态

```bash
openclaw status
```

#### 步骤 2：如果 Gateway 没起来，就重启

```bash
openclaw gateway restart
```

#### 步骤 3：再看一次状态

```bash
openclaw status
```

如果第二次状态正常，说明：
- 不是永久性故障
- 至少核心服务恢复了

如果还不正常，再往下查。

---

## 一个最常见场景：你以为它坏了，其实只是入口没通

比如：
- Telegram 没回消息
- 你以为整个 OpenClaw 挂了

但一看 `openclaw status`，Gateway 明明是好的。
那这时候就别把锅全甩给系统整体。
更可能是：
- 某个消息渠道异常
- 策略限制
- token/认证问题
- 网络问题

这就是为什么状态比脑补重要。

---

## 今天推荐你实际做一遍

### 演练 1：看状态

```bash
openclaw status
```

### 演练 2：看 gateway 专项状态

```bash
openclaw gateway status
```

### 演练 3：做一次安全重启

如果你当前环境允许、并且你知道此时重启不会打断关键操作，可以执行：

```bash
openclaw gateway restart
```

然后再看：

```bash
openclaw status
```

这样你就不是“知道命令”，而是真的走过完整闭环了。

## 怎么判断算“基础正常”

你可以先用一个粗暴标准：

- Gateway 正常运行
- 没有明显致命报错
- 你关心的入口没集体炸

如果满足这三条，至少说明：
**系统主干基本还活着。**

---

## 今天最容易踩的坑

### 坑 1：不看状态，直接改配置

这是非常经典的作死姿势。
很多问题跟配置压根没关系。

### 坑 2：重启了但不复查

你重启只是动作，
**复查状态才是验证。**

### 坑 3：把局部异常当成整体死亡

比如 Telegram 不通，不代表整个 OpenClaw 就死透了。
要分层看。

### 坑 4：一看到报错就慌

先判断：
- 是 warning 还是 fatal
- 是局部问题还是全局问题
- 是暂时问题还是持续问题

别还没看懂就开始乱挥刀。
