# Day 2：启动、状态与基础排障 —— 先确认机器活着，再谈学习

## 先讲人话：今天在学什么

今天不是修大故障，今天学的是：

> **当 OpenClaw 看起来不对劲时，你第一步该看哪里。**

这很重要，因为很多人一出问题就开始乱改配置，最后把“小毛病”修成“火葬场”。

---

## 这节课的核心意思

OpenClaw 出问题时，最常见的不是“系统彻底废了”，而是下面这几类：
- Gateway 没起来
- Gateway 起了，但某个组件状态不对
- 你以为它坏了，其实只是没看状态

所以 Day 2 的核心，不是让你会多少命令，而是让你形成一个习惯：

**先看状态，再动手。**

---

## 今天你要学会什么

1. 会跑 `openclaw status`
2. 知道输出里大概该看什么
3. 会启动 / 重启 Gateway
4. 知道遇到异常时，先做哪几步基础排查

---

## 今天怎么学

### 第一步：看系统状态

执行：

```bash
openclaw status
```

先别被输出吓到。
你只需要盯这几个问题：

- 有没有报错
- Gateway 是不是正常
- 有没有明显提示某个模块不健康

你可以把 `status` 理解成体检单。
不是所有指标都要第一天看懂，但至少你得知道：**不舒服先看体检单。**

### 第二步：如果状态不正常，尝试重启 Gateway

执行：

```bash
openclaw gateway restart
```

然后再看一次：

```bash
openclaw status
```

这一步的目的不是迷信重启，而是验证：
**问题是“暂时卡住”，还是“配置/环境层面的持续故障”。**

### 第三步：如果还是不对，至少先分层

你现在先学会这种判断方式：

- `status` 就报错：先怀疑系统层 / 配置层
- Gateway 起不来：先怀疑配置或依赖
- Gateway 正常，但某功能不通：再看对应模块

先分层，再排障。别一把梭全改。

---

## 一个很实用的人话心法

以后碰到 OpenClaw 相关问题，脑子里先过这一句：

> “它到底是没启动、没接通、还是我没看对地方？”

这句话能帮你避开很多低级坑。

---

## 今天最容易踩的坑

### 坑 1：看到问题先改配置
错。先看状态，不然你是在盲修。

### 坑 2：重启完不复查
你如果不再跑一次 `status`，那重启等于白重启。

### 坑 3：把局部问题当全局崩盘
比如 Telegram 不通，不一定是整个 OpenClaw 挂了。

---

## 今天的验收标准

做完后，你至少要能告诉我：

1. `openclaw status` 现在是否正常
2. 如果不正常，异常主要落在哪一层
3. 你有没有尝试过 `openclaw gateway restart`
4. 重启后状态有没有变化

---

## 做完后怎么回我

你直接发：

1. `openclaw status` 的总体结论（正常 / 有异常）
2. 你看到的最关键一条状态信息
3. 如果你重启了，重启前后有什么变化
