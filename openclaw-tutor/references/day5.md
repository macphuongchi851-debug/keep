# Day 5：配置文件基础 —— 先学会读和校验，别上来就瞎改

## 今天学什么

配置文件像 OpenClaw 的“系统总设置”。
你后面想接 Telegram、改模型、调浏览器、动自动化，基本都绕不过它。

今天不讲骚操作，只讲 3 件最要命的基本功：

1. 找到配置
2. 读懂局部配置
3. 修改前先校验和备份意识

---

## 为什么今天不能直接教你乱改

因为很多人一看到配置文件就走两个极端：

- 不敢碰
- 瞎鸡巴改

这两种都不行。

你真正需要的能力是：

> **先学会安全地看，再学会安全地改。**

---

## 今天你要知道的 4 件事

1. 配置文件在哪类位置
2. 为什么不要靠肉眼乱翻整份大 JSON
3. 如何只看你关心的配置项
4. 改配置前为什么要先备份、先校验

---

## 配置为什么这么重要

OpenClaw 里很多关键能力都靠配置控制：
- gateway 行为
- 模型组织
- 渠道开关
- 浏览器/插件参数
- session 边界
- 消息策略

所以配置文件不是“附属品”，
而是系统行为说明书。

---

## 今天怎么学

### 第一步：确认配置不是随便乱猜字段名

最忌讳的就是：
- 凭记忆猜字段
- 在网上看到旧版本配置就照抄
- 一次性把整份文件乱翻到眼花

更稳的思路是：
- 先知道自己要查哪一段
- 尽量查具体路径
- 查 schema 或当前值，而不是纯脑补

---

### 第二步：先看“局部”，不要一上来硬啃整份文件

如果你想看某个配置项，优先看具体路径。
比如你想查模型，就看模型相关子树；
想查 Telegram，就看 Telegram 那段；
想查 session，就看 session 那段。

这样有两个好处：
- 不容易看花
- 不容易改错地方

---

### 第三步：修改前，先备份

这是硬规矩，不是建议。

尤其是像 `~/.openclaw/openclaw.json` 这种核心配置，
每次改之前都应该：

1. 先备份一份带时间戳的副本
2. 再修改
3. 再做 JSON / 配置校验
4. 最后才应用或重启

为什么？
因为一旦改坏：
- 入口可能断
- Gateway 可能起不来
- 你可能自己把自己锁外面

别觉得这是多余步骤。
这叫留后路。

---

### 第四步：校验不是形式主义

很多人以为“文件能保存”就等于没问题。
错。

常见翻车方式：
- JSON 格式错
- 字段类型错
- 路径写错
- 结构层级错
- 当前版本根本不认这个字段

所以改完一定要验证：
- 格式是否正确
- 字段是否在 schema 允许范围内
- 应用后系统是否正常

---

## 一个正确的配置操作流程

你以后默认按这套来：

1. 明确目标：我要改什么
2. 查 schema / 查当前值
3. 备份配置文件
4. 修改局部配置
5. 校验格式和结构
6. 应用 / 重启相关服务
7. 再看状态确认真的生效

照这个流程，你出事概率会低很多。

---

## 今天建议你做的最小练习

### 练习 1：确认你知道配置文件在哪

你不一定今天就去改，
但你至少要知道核心配置大概在哪里、谁在控制它。

### 练习 2：只查一个具体配置项

比如：
- 当前模型相关项
- 某个 channel 开关
- session 相关配置

重点是学会“定点看”，不是“全文件乱翻”。

### 练习 3：说出为什么备份不能省

这个非常关键。
你能讲清楚原因，说明你已经开始有配置安全意识了。

---

## 今天最容易踩的坑

### 坑 1：照着旧教程乱填字段

OpenClaw 版本会变，旧字段不一定还能用。

### 坑 2：改之前不备份

这属于自断后路。

### 坑 3：把“保存成功”当成“配置正确”

保存只是写进去了，不代表能用。

### 坑 4：一上来改很多项

真出问题时你都不知道是哪一刀捅死的。

---

## 今天的验收标准

你做完后，要能回答：

1. 为什么配置要“局部看、局部改”
2. 为什么改前必须备份
3. 为什么 schema / 校验很重要
4. 如果改完系统异常，你第一步做什么

---

## 做完后怎么回我

你直接回：

1. 你今天打算优先关注哪个配置区域
2. 为什么不该乱翻整份大配置
3. 你理解的“先备份再改”到底是在防什么
4. 你以后改配置的标准流程是什么

## 课后一句话总结

配置文件不是不能碰，
是不能像拆炸弹一样闭眼乱剪。
