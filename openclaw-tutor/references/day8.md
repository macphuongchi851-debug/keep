# Day 8：Telegram 接入（下）—— 真接上、真发消息、真排故障

## 先讲人话：今天在学什么

昨天你已经搞明白 Bot、Token、访问边界是啥。
今天才是真正落地：

> **把 Telegram 接进 OpenClaw，让它能收你的消息、回你的消息。**

这一节学完，你的 OpenClaw 才算从“本地能用”进化到“手机上也能叫得动”。

---

## 今天要搞明白的 3 件事

1. Token 负责“接得上”
2. 策略负责“谁能进来”
3. Gateway 状态决定“系统是不是真的活着”

你后面排障，其实就是围着这三件事转。

---

## 今天怎么学

### 第一步：把 Telegram 配进 OpenClaw

你需要把以下思路落到配置里：

- 开启 Telegram 渠道
- 填入 bot token
- 选好私聊策略
- 如果是 allowlist，就把允许的用户 ID 放进去

今天别死背整段配置模板。
你先记住逻辑：

**Token 决定接不接得上，策略决定让不让你进门。**

### 第二步：重启 Gateway

配置变更后，执行：

```bash
openclaw gateway restart
```

然后立刻看状态：

```bash
openclaw status
```

你现在要观察的是：
- Gateway 是否正常回来
- Telegram 相关状态有没有直接报错

### 第三步：去 Telegram 私聊这个 Bot

你发一句最简单的话就够：

```text
你好，测试一下。
```

如果走的是 `pairing`，你可能要先完成批准流程。
如果走的是 `allowlist`，那就要确认你的 Telegram 用户 ID 已经在允许列表里。

### 第四步：如果失败，按这三个方向排

#### 1）像是 token 问题
常见表现：
- unauthorized
- 无法认证
- Telegram 直接报连接/鉴权失败

#### 2）像是访问策略问题
常见表现：
- Bot 在
- 你也发得出去
- 但消息进不了实际会话

#### 3）像是 Gateway 问题
常见表现：
- 整体状态就不正常
- Telegram 只是一起陪葬

---

## 今天你真正要学到的判断力

学完今天，不是只会“配成功一次”，而是脑子里能形成这个排障框架：

> 接不上，是 token 问题？
> 进不去，是策略问题？
> 全都不对，是 Gateway 问题？

有这个判断框架，你后面就不容易乱撞墙。

---

## 最容易踩的坑

### 坑 1：配完不重启就开始测试
那很多变化根本没生效。

### 坑 2：只测消息，不看状态
这样排障会特别瞎。

### 坑 3：把策略问题误判成 token 问题
Bot 在但你进不去，不一定是 token 错。
