# 输入要求与业务规则

## 1. 输入文件要求

### 店铺订单表
至少应包含字段：
- 商品
- 订单号
- 订单状态
- 商家实收金额(元)
- 商品数量(件)
- 发货时间

### 供应商发货表
至少应包含字段：
- 店铺
- 日期
- 订单号
- 品名
- 数量
- 代发价格
- 快递附加运费

### 广告数据表
至少应包含字段：
- 商品名称
- 总花费(元)
- 交易额(元)
- 净交易额(元)

---

## 2. 平台筛选规则

只保留标题包含以下关键词的订单：
- 微景观
- 苔藓

按订单号聚合。

### 平台件数相关定义
- 平台下单件数：筛选后全部订单件数
- 有效销售件数：剔除退款 / 取消后件数
- 平台已发货件数：**主看订单状态**，状态显示 `已发货` / `已收货` / `已发货退款成功` / `已发货，售后处理中` 的件数
- **`已取消` 订单即使残留发货时间，也不得计入平台已发货件数**
- **`未发货，退款成功` 即使有发货时间，也不得计入平台已发货件数**
- **货品成本 = 0 的订单，不���响平台已发货件数统计**（2026-03-21修正）；货品成本=0的限制只用于平台发货总成本计算，不用���已发货件数统计

## 3. 供应商清洗规则

### 必须处理的脏数据

#### 3.1 历史月份混入
供应商表可能混入别的月份数据，但**当前最新版 2.3 默认不再因为目标周日期把用户给定的供应商发货硬过滤掉**。除非用户明确要求按日期裁切，否则按用户提供这批供应商数据整盘计算。

**2026-03-23 再次焊死：** 这个“整盘计算”不仅适用于 `供应商表格原始总件数` / `发货表格总成本`，也适用于**供应商明细侧**（如 `供应商有、平台无`、`订单匹配表`、`最终异常订单清单`）。默认**不得**再因为供应商文件名周期或行内日期不在目标周，就把这些供应商订单从明细里二次裁掉；否则会出现“总表保留了，明细丢了”的口径撕裂。

### 3.1.1 供应商来源文件归属优先级（2026-03-15 固化）
当供应商表内“店铺”列填写不稳定、混写、缺失，或同一文件内出现多种店铺名时：
- 允许在 `rules.store_normalization.supplier_source_file_store_map` 中按**供应商文件名**直接指定归属店铺
- **来源文件归属只允许用于补全空/缺失店铺，不允许整表强行覆盖已明确写出的店铺值**
- 典型固化案例：
  - `秘牌1店供应商发货` → `秘牌宠物旗舰店`
  - `秘牌2店供应商发货` → `秘牌2店`
- 对应平台归属再通过 `platform_supplier_store_map` 映射，例如：
  - `秘牌旗舰店` → `秘牌宠物旗舰店`
  - `秘牌小店` → `秘牌2店`
  - `小须鲸旗舰店` → `小须鲸`
  - `金鱼旗舰店` → `金鱼`

结论：**发货表格2对应的发货默认归属秘牌小店，不得再挂到秘牌旗舰店；但如果表内已明确写了别的店铺，也不能被文件名整表硬覆盖。**

#### 3.2 一单多行
同一订单可能拆成：
- 主商品
- 配件 / 灯
- 附加运费

应按订单号聚合。

#### 3.3 空订单号补充行
只有明显属于上一单的补充行，才允许继承上一订单号。

**强制补充规则（2026-03-14新增，必须死记）：**
- **不能因为空订单号那行“有品名 / 有价格”，就直接继承上一单订单号**
- 如果这一行本身像一条**独立发货记录**（例如：地址变化、快递单号独立、商品独立成行），就必须视为**单独一行待核对记录**，**不得续接上一单**
- 典型反例：上一行是订单 `260305-007046510971691`，下一行空订单号但有 `普通植物灯`、独立快递单号、独立地址，这一行**不能并到上一单**
- 结论：**“空订单号 + 有品名/价格” 不是继承条件，只能算“疑似补充行”，必须继续核独立性**

#### 3.3.1 空店铺补充规则（2026-03-21新增，晚口径修正）
- **空店铺不等于默认继承上一行店铺**
- 只有在**同一订单号的补充行**场景下，才允许补继上一行店铺
- 如果是**空订单号 + 空店铺**，但本身有地址 / 快递单号 / 日期等独立特征，说明它是一条**独立发货记录**，则：
  - **不得继承上一行店铺**
  - **不得强挂到来源文件默认店铺**
  - **按用户最新口径：这类记录不计入供应商平台匹配件数，也不挂任何店铺**
- 典型反例：`定时植物灯` 单独一行、店铺空、订单号空、快递单号独立，这种不能因为它出现在 `秘牌2店供应商发货.xlsx` 里，就直接挂成 `秘牌2店`；按最新口径应直接**不计入**

#### 3.4 表尾合计行 / 空白污染行
必须跳过，不能续接到上一订单。

典型特征：
- 无日期
- 无品名
- 无订单号
- 快递单号位置出现“合计”
- 只有金额，没有订单信息

这类行如果算进去，会把利润直接算炸。

## 4. 成本定义

### 单订单供应商总成本
= 代发价格

**强制说明（2026-03-15固化）：**
- 供应商表中的“快递附加运费”列按**备注列**处理
- 不论里面写 `运费+1`、`礼盒+3`、`这单算30` 还是其他文字，**一律不计入成本**
- **2026-03-23 补充显示规则：** 如果供应商原表里的 `代发价格` 本身就是负数，则相关明细表中的 `货品成本 / 总成本 / 供应商总成本` 也应**原样显示负数**；默认**不得**自动转正、取绝对值或做美化替换

### 平台发货总成本
= 平台已发货 / 已收货 / 已发货退款成功，且**订单号匹配成功并且件数一致**订单的供应商**货品成本**汇总

**注意：**
- **不计快递附加运费**
- **订单号一致但件数不一致，不算匹配成功**
- 供应商空订单号独立记录不得错误续接到上一单，否则会把成本算大

### 发货表格总成本
= 用户提供的这批供应商发货原表中的货品成本汇总

**2026-03-16 再次焊死：**
- **发货表格总成本 / 供应商原始侧总成本，默认不得按目标周日期二次过滤**
- 只要是用户这次交来的供应商原表中的有效发货行，就应保留在原始总成本口径里
- 除非用户明确要求“按日期裁切供应商原表”，否则脚本、规则、回复都不得私自按文件名周期或目标周日期删行

### 发货表格件数 / 供应商件数（强制区分）
- **两个供应商表必须一起看、一起算**
- **供应商表格原始总件数**：两个供应商表中，这一批次原始表格里的实际发货件数合计，用于解释“表上为什么看到这么多件”“付款给别人多少件”
- **供应商表格原始总件数默认不得按目标周日期二次过滤**；除非用户明确要求，否则不能因为文件名周期、目标周起止日期、或行内日期不在周期内，就把原表有效行踢掉
- **供应商平台匹配件数**：本次结算实际采用的、且**供应商订单号能匹配到平台订单号**的发货件数
- **字段名说明**：原来的“供应商纳入结算件数”现统一改名为 **`供应商平台匹配件数`**
- **供应商订单号匹配不上平台订单号的，不计入供应商平台匹配件数**
- **空订单号行不计入供应商平台匹配件数**
- **空店铺独立发货行也不计入供应商平台匹配件数**
- 对外回复必须说清是哪一个，不能混口径
- 如果用户明确说“表格里有多少件”“付款给别人多少件”，优先先报**供应商表格原始总件数**，再补充**供应商平台匹配件数**
- 结果表建议同时展示这两个件数，避免复盘时再次吵口径
- **供应商与平台发货件数差**：固定定义为 **供应商表格原始件数 - 供应商平台匹配件数**，用于展示供应商原表里存在、但未进入平台匹配口径的件数
- **重点展示建议（2026-03-21 晚新增）**：`供应商表格原始总件数`、`供应商平台匹配件数`、`供应商与平台发货件数差` 三个字段建议在总表中使用不同但协调的重点色显示，方便快速定位关键差异

## 5. 收入与利润定义

### 平台收入
= 商家实收金额汇总，剔除：
- 未发货退款成功
- 已取消
- 已发货退款成功

### 平台利润
= 平台收入 - 平台发货总成本 - 平台佣金 - 广告费

说明：
- **货品率指标已取消**
- 保留 **成本率**、**广告费率** 作为经营分析指标
- 可额外输出 **“利润率低于17%的订单”** 子表，用于筛出订单级低利润单
- 该子表应包含：平台店铺、订单号、商品标题、**商品规格**、订单状态、件数、订单收入、货品成本、订单佣金、订单利润、订单利润率

### 平台佣金
= 佣金基数 × 0.6%

当前默认佣金口径：
- 只要是拼多多相关出单，就计入佣金基数
- **退款单也计佣金**
- 默认仅剔除：
  - 已取消
  - 待付款

也就是说，佣金不再简单跟随“平台收入”口径走。
默认更接近：
- **拼多多微景观 / 苔藓相关订单（剔除已取消、待付款）× 0.6%**

## 6. 异常订单定义

### 供应商有、平台无
供应商表有订单号，但平台筛选结果无该订单号。

**注意：** 这类单即使没有平台匹配，仍然属于供应商实际付款/实际发货的一部分；在“供应商表格原始总件数”里必须保留。

**2026-03-23 固化：** 只要这笔订单属于**用户本次提供的供应商原表中的有效订单号记录**，默认就应进入 `供应商有、平台无` 明细；**不得**因为该行日期早于/晚于目标周，或供应商文件名写的是某个周期，就把它从这个子表里再次裁掉。除非用户明确要求“明细也按日期裁切”，否则统一按整批原表口径展示。

### 平台有、供应商无-全部
平台筛选后有订单号，但供应商无匹配。

### 平台有、供应商无-退款取消
订单状态为：
- 未发货，退款成功
- 已取消

这类不算真正发货异常。

### 待发货未匹配
订单状态为：
- 待发货

这类单独列，不计入最终异常。

### 最终异常订单
满足以下条件：
- 平台已发货 / 已收货 / 已发货退款成功
- 供应商无订单号匹配记录

这类才是需要重点追查的异常。

### 额外专项子表（2026-03-15新增）
- **重复发货**：供应商同订单号发货件数 > 平台件数
- **最终异常订单清单（2026-03-21 晚最新口径）**：不再展示旧的“平台应发异常单”，改为展示 **“供应商与平台发货件数差”** 的构成明细。也就是：凡是计入“供应商表格原始件数”、但**没有进入“供应商平台匹配件数”** 的订单/原始行，都应在这张表展示，并逐行注明原因。

## 7. 小须鲸专项规则

必须单独输出：
- 平台微景观总件数
- 平台有效销售件数
- 平台已发货件数
- 供应商发货件数
- 匹配成功订单数
- 平台有、供应商无件数
- 供应商有、平台无件数
- 订单明细清单

---

## 8. 建议核对顺序

生成文件后，先核：
1. 总表利润是否离谱
2. 供应商成本构成明细是否出现异常大单
3. 最终异常订单清单是否误混退款/取消
4. 待发货未匹配是否已单独拎出
5. 小须鲸专项是否对得上
6. 老板汇报版文字是否和表格一致
