# 微景观结算 2.3

用于**周度微景观 / 苔藓对账**，目标是产出**最终可交付 Excel 文件**，不是口头估算。

这版重点解决 4 个老毛病：
- **不再把某一周数据硬编码进脚本**
- **口径规则外置，可配置**
- **技能说明、规则文件、执行脚本同源**

当前版本定义：
- **2.0** = 完成规则、skill、脚本一体化重构
- **2.1 / 2.2 / 2.3** = 在 2.0 框架上继续做校验与口径修正迭代
- **当前最新版对外口径统一按 2.3**

## 什么时候用

当用户要做以下事情时触发：
- 微景观 / 苔藓业务周度结算
- 店铺数据与供应商发货数据核对
- 广告费归集
- 利润核算
- 异常订单排查
- 小须鲸店铺专项核对
- 生成老板看的汇报版表格
- 回溯某一周口径并整盘重算

这版新增两块：
- **店铺映射外置**：广告店铺名、平台店铺名、供应商店铺名、供应商来源文件→店铺归属、平台→供应商映射都可以写进 rules
- **踩坑回溯固化（2026-03-15）**：文件名周期清洗、`已取消` 不计平台已发货、`秘牌2/秘牌2店` 统一映射、供应商“表格原始总件数”和“周内有效发货件数”必须分开讲、`秘牌2店供应商发货` 默认归属秘牌小店，禁止混口径回复
- **2026-03-16 再焊一层**：供应商“表格原始总件数”与“发货表格总成本/供应商原始侧总成本”默认都按**用户本次提供的供应商原表整批统计**，除非用户明确要求，否则**不得**按目标周日期、文件名周期、或行内日期再做二次过滤
- **2026-03-21 再补一刀**：总表固定增加 `未知店铺` 行，用于承接历史上供应商表里店铺为空/缺失的数据口径回溯；但**用户最新口径（2026-03-21 晚）**已改为：**空订单号行一律不计入“供应商平台匹配件数”，空店铺独立发货行也一律不计入“供应商平台匹配件数”**；同时，**供应商订单号匹配不上平台订单号的，也不计入“供应商平台匹配件数”**
- **字段口径同步（2026-03-21 晚）**：原“供应商纳入结算件数”统一更名为 **`供应商平台匹配件数`**。它的定义不再是“用户给了什么就算什么”，而是：**供应商订单号能匹配到平台订单号** 的件数；匹配不上平台的供应商订单、空订单号行、空店铺独立发货行，统统不计入这个字段
- **2026-03-23 补焊口径一致性**：供应商**明细侧**（如 `供应商有、平台无`、订单匹配、异常明细）默认也必须按**用户本次提供的整批供应商原表**参与匹配与展示，**不得**再因为供应商文件名周期或行内日期不在目标周，就把这些订单二次裁掉；否则会出现“供应商表格原始总件数保留了，但供应商有/平台无明细没了”的口径撕裂
- **2026-03-23 负数显示规则**：如果供应商原表里的 `代发价格` / 汇总 `总成本` / `供应商结算成本` 本身就是负数（如原表存在负数冲销、回退、红字记录），默认**按原表原样显示负数**；**不得**自动取绝对值、改成正数、加括号美化，避免把原始财务语义洗掉
- **差异字段同步（2026-03-21 晚）**：原“已发货-供应商平台匹配件数差”统一更名为 **`供应商与平台发货件数差`**，计算逻辑固定为：**供应商表格原始件数 - 供应商平台匹配件数**；不再使用平台已发货件数参与这个字段计算
- **硬拦截焊死（2026-03-15 晚）**：供应商原始总件数、各店拆分件数、逐文件件数、平台→供应商店铺映射，四层任一对不上直接失败，不准出表；如用户启用“空订单号/空店铺独立行不计入”新口径，则这些行必须在清洗阶段直接剔除，不能再混入 summary

**原则：**
- 改口径，优先改 `rules.json`
- 改通用逻辑，改 `build_settlement.py`
- 改完必须重新生成并打开文件复核

## 默认口径（由 rules 驱动）

默认规则文件：`references/default_rules.json`

### 1. 平台标题筛选
只保留标题包含以下关键词的订单：
- 微景观
- 苔藓

### 2. 平台收入口径
默认**剔除**以下状态关键词（订单状态 + 售后状态双重判断）：
- 退款成功
- 退款中
- 售后处理中
- 已���消
- 待发货
- 待付款

**重要：** 判断时同时检查原始表的「订单状态」和「售后状态」两列���只要任意一列含以上关键词，该订单一律不计入平台收入。不能只看订单状态，漏掉售后状态为���售后处理中」或「退款成功」但订单状态仍显示「已发货，待收货」的单。

并且默认要求：
- 必须有**有效货品成本匹配**
- 默认剔除**货品成本为 0** 的单

### 2.1 平台佣金口径
默认按用户最新确认口径执行：
- **只要是拼多多相关出单，就计佣金**
- **退款单也计佣金**
- 默认仅剔除：`已取消`、`待付款`

也就是说：

**平台佣金 = 拼多多微景观/苔藓相关订单（剔除已取消、待付款）的佣金基数 × 0.6%**

注意：
- **佣金口径独立于平台收入口径**
- 不能再用“剔除退款后的平台收入 × 0.6%”代替佣金

### 3. 平台发货相关口径

#### 平台已发货件数
默认按用户最新确认口径执行：
- **主看订单状态，不再让发货时间单独触发平台已发货判断**
- 状态含`已发货` / `已收货` / `已发货，退款成功` / `已发货，售后处理中` 的，才有资格算平台已发货
- **`已取消` 一律不得算平台已发货**
- **`未发货，退款成功` 不算平台已发货**，哪怕残留发货时间也不算
- **货品成本 = 0 的订单，不计入平台已发货件数**（除非用户另行指定）
- 也就是说，哪怕状态是 `已发货，待收货` / `已收货`，只要货品成本 = 0，也一律不算平台已发货

也就是说：

**平台已发货件数 = 订单状态明确属于已发货语义的订单件数，但不包含 `已取消`、`未发货退款成功`，且默认不包含货品成本为 0 的订单。**

#### 发货表格件数
默认按用户最新确认口径执行：
- **两个供应商表必须一起算**，禁止只盯其中一个表回复用户
- **按用户提供的数据整盘算，不再因为目标周日期把供应商发货硬过滤掉**
- **供应商明细侧也必须跟这个口径一致**：`供应商有、平台无`、`订单匹配表`、`最终异常订单清单` 这类明细表，默认都不得再按文件名周期/行内日期把供应商订单二次裁掉
- **有订单号**的记录正常按订单聚合
- **空订单号行按 2026-03-21 晚口径一律不计入供应商平台匹配件数**
- **供应商订单号匹配不上平台订单号的，也不计入供应商平台匹配件数**

**强制区分两个概念（2026-03-15 回溯新增，2026-03-21 晚口径修正）：**
- **供应商表格原始总件数**：两个供应商表里，这一批次表格中出现的原始实际发货件数合计，用于解释“表格里明明有多少件”“付款给别人一共多少件”
- **供应商平台匹配件数**：本次结算实际采用的、且供应商订单号能匹配到平台订单号的发货件数

对外回复时，**必须明确自己说的是哪个件数**，不能把不同口径混成一个数。

并且：
- **总表必须同时展示这两个件数**
- 用户质疑供应商件数时，必须回到**两个供应商原始表逐表核对**

也就是说：

**不续接上一单，不等于不计入发货件数。也不能再私自按日期把用户给的供应商数据裁掉。**

#### 平台发货总成本
默认：
- 必须**订单号匹配成功**
- 必须**平台件数 = 供应商件数**
- 默认只统计**货品成本**
- 默认**不计快递附加运费**
- **供应商“快递附加运费”列按备注列处理，整列不得参与任何成本计算**
- 默认只统计**已发货 / 已收货**口径订单

### 4. 空订单号补充行规则
默认：
- 空订单号行**不能**因为“有品名 / 有价格”就自动续接上一单
- 如果带有独立特征（地址、快递单号、日期等），必须视为独立待核对记录
- 不能把独立记录错并到上一单

### 5. 特例订单规则
不再把某周订单号写死在脚本里。

统一放到规则文件：
- `platform_income.special_include_orders`
- `platform_income.special_exclude_orders`

### 6. 校验规则
可在规则文件 `checks.expected_metrics` 里写入本批次校验值，例如：
- `total_income`
- `total_platform_cost`
- `total_ad_spend`
- `total_profit`
- `real_abnormal_count`

严格模式下，关键预期值不符会直接报错，防止错表流出去。

## 硬规则

- 做一步，落一步
- 改规则 → 改脚本（如需要）→ 重跑 → 打开 sheet 核对 → 再回复用户
- 没核完，不准说“改好了”
- 用户要的是**最终文件正确**，不是解释
- 一个口径变动，相关表头、公式、明细、摘要、异常分类都必须联动检查
- **禁止**用“旧结果 ± 差额”偷懒
- 必须从**原始平台单 + 原始供应商单 + 原始广告单**整盘重算

## 标准流程

1. **确认输入完整**
   - 店铺订单导出表
   - 供应商发货表
   - 广告数据表
   - 通常来自同一个 zip

2. **确认本周是否有特殊口径**
   重点问清：
   - 待发货是否剔除
   - 退款中 / 售后处理中是否剔除
   - 退款成功 / 已取消是否剔除
   - 快递附加运费是否计入
   - 是否有“本周下单、下周发货”特例单
   - 是否有必须强制纳入 / 剔除的订单号

3. **准备规则文件**
   - 无特殊情况：用 `default_rules.json`
   - 但如果 `references/` 下存在与本批次周期同名的周期规则文件（如 `example_rules_2026-03-02_2026-03-08.json`），脚本必须自动叠加该文件，不能只吃默认规则
   - 有特殊情况：复制一份规则文件按本周改
   - 历史案例可参考：`example_rules_2026-03-02_2026-03-08.json`

4. **运行脚本整盘生成**
   默认命令：

```bash
python3 weijingguan-settlement-v2-3/scripts/build_settlement.py \
  --zip "/path/to/input.zip" \
  --output "/path/to/output.xlsx" \
  --start 2026-03-02 \
  --end 2026-03-08 \
  --rules "/path/to/rules.json"
```

如只想先看结果、不想因校验失败中断：

```bash
python3 weijingguan-settlement-v2-3/scripts/build_settlement.py \
  --zip "/path/to/input.zip" \
  --output "/path/to/output.xlsx" \
  --start 2026-03-02 \
  --end 2026-03-08 \
  --rules "/path/to/rules.json" \
  --no-strict-checks
```

5. **打开结果文件核对**
   必核：
   - 总表（尤其核“平台已发货件数”是否已扣除货品成本为0订单）
   - 老板汇报版
   - 最终异常订单清单
   - 小须鲸专项核对
   - 总表下方的运营分析文字区

6. **发现问题时**
   - 先判断是规则问题还是脚本通用逻辑问题
   - 规则问题：优先改 rules
   - 通用逻辑问题：改脚本
   - 如果本周已有周期规则文件，执行时必须确认脚本实际加载到了该文件；不能口头说“规则已经有了”，结果运行时没带上
   - 然后**整盘重算**
   - 重新打开关键 sheet 复核
   - 只要出现“原始表件数”和“总表件数”打架，先停下来修规则/脚本，**禁止先把错表发给用户**

## 输出文件至少包含

- 总表
- 老板汇报版

### 关于“小须鲸专项核对”（2026-03-21 晚最新口径）
- 这张表应**参考总表逻辑**输出，而不是保留历史旧专项样式
- 默认只保留：
  - 平台下单件数
  - 有效销售件数
  - 平台已发货件数
  - 供应商表格原始总件数
  - 供应商平台匹配件数
  - 供应商与平台发货件数差
  - 平台发货总成本
  - 发货表格总成本
  - 平台收入
  - 佣金基数
  - 广告费
  - 平台佣金
  - 平台利润
  - 退款/成本率相关指标
- **不再默认展示旧版订单级长明细**；除非用户明确要求，再单独加明细

### 关于“最终异常订单清单”（2026-03-21 晚最新口径）
- 这张表**不再沿用旧的“平台应发异常单”口径**
- 这张表现在用于承接 **“供应商与平台发货件数差”** 的详细构成明细
- 默认应包含：
  - 序号
  - 归属店铺
  - 供应商店铺
  - 订单号
  - 日期
  - 商品/品名
  - 件数
  - 供应商总成本
  - 原因备注
- 每一行都必须写清楚为什么进入差异：
  - 供应商订单号匹配不上平台订单号
  - 空订单号
  - 空店铺
  - 或其他导致未进入“供应商平台匹配件数”口径的原因

### 重点字段高亮（2026-03-21 晚新增）
- 在 **总表** 中，以下三个字段必须使用**不同但协调的重点色**高亮，便于老板/操盘手快速扫数：
  - `供应商表格原始总件数`：柔和杏橘色
  - `供应商平台匹配件数`：柔和薄荷绿色
  - `供应商与平台发货件数差`：柔和淡紫色
- 要求：
  - **表头**高亮
  - **整列数据**高亮
  - **总计行**保持同色系，不允许被默认样式覆盖
- 在 **老板汇报版** 中，与这三项相关的摘要句允许做浅色重点提示，但必须保持整体美观，禁止使用刺眼荧光色

- 平台有、供应商无不能把退款 / 取消 / 退款中全算成异常
- 待发货不能混入最终异常

- 件数不一致是否没有误判为匹配成功
- 空订单号独立记录是否没有被错误续接
- 货品率指标已移除后，相关表头/摘要/分析文案是否已同步清干净
- 老板汇报版文案是否和数字一致
 sheet 是否通过预期检查
- 如果本轮改过口径，是否已经整盘重算
